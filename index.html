Slide 8 Core setup Testable framing

The lecture begins with the right question: how do we turn an image into the kind of object a language model can use?

The slide says that text models already operate on a sequence of vectors, so a multimodal system needs an image encoder \(f_{\mathrm{enc}}(x_{\mathrm{image}}) \rightarrow z_1, \dots, z_L\). This is not just notation. It is the design problem that determines almost everything downstream: what the tokens mean, how much spatial detail survives, how expensive the model is, and how easily the model can share information across vision and language.

Why this is important

Without a learned image representation, a language model has no principled way to map a two-dimensional pixel grid into the one-dimensional token stream it expects.

Intuition

Text arrives already chunked into tokens. Images do not. Multimodal modeling starts by inventing the visual analogue of tokens.

Missing context

Image “tokens” are usually continuous vectors, not discrete vocabulary items. They also need some positional signal, because a flattened sequence otherwise loses the image’s spatial layout.

Exam takeaway

If a question asks what multimodal modeling adds beyond a plain LM, the clean answer is: a vision representation architecture plus a learning objective that makes those vectors useful.

Slides 9-16 ViT mechanics Quiz-aligned

ViT solves the representation problem by treating image patches like tokens

ViT divides an image \(x_{\mathrm{image}} \in \mathbb{R}^{H \times W \times C}\) into \(N = \frac{HW}{P^2}\) square patches of side length \(P\). Each patch is flattened into a vector in \(\mathbb{R}^{P^2 C}\), linearly projected into dimension \(D\), and then combined with position embeddings. After that, the model is just a transformer operating on a sequence of visual token vectors.

The two formulas worth memorizing are the token count and the scaling implication.

\[ N = \frac{HW}{P^2} \qquad\text{and}\qquad \text{self-attention cost} \propto N^2. \]

If you halve the patch size while keeping image resolution fixed, the number of tokens becomes \(4N\), so the attention cost becomes \(16\times\) larger. That exact reasoning already appears in the quiz set.

What the slides are saying

Patchification makes images compatible with transformer machinery, while self-attention lets the model combine information from different parts of the image before classification or downstream use.

Why it works

Each patch starts as a local snapshot, but the transformer can build global context by letting each patch attend to others. That is why ViT can identify salient regions instead of relying only on local convolutions.

What the figure-heavy slides add

The later ViT slides show three qualitative facts: learned position embeddings preserve spatial regularity, attention highlights task-relevant regions, and deeper layers tend to aggregate information over larger spatial spans.

Common misunderstanding

Patch size is not a harmless implementation detail. Smaller patches preserve finer detail, but they explode token count and compute.

Slides 20-24, 26, 28 CLIP Very likely tested

CLIP learns image representations from language by aligning matching image-text pairs in a shared space

The lecture motivates CLIP by contrasting it with older image pretraining that used only class labels. A caption carries much richer information than a single class name, and web-scale image-text pairs are easier to scale than hand-labeled classification datasets. CLIP therefore learns an image encoder \(f_I(x)\) and a text encoder \(f_T(y)\) whose outputs have high similarity when the image and caption match and low similarity when they do not.

The key idea is easier to remember in words than in symbols:

\[ \text{raise } f_I(x_n)^\top f_T(y_n) \quad\text{and lower}\quad f_I(x_j)^\top f_T(y_n),\; f_I(x_n)^\top f_T(y_j) \]

for mismatched pairs in the same batch. The displayed loss is just two cross-entropy terms, one softmaxing over images and one softmaxing over text.

Why it matters

Once images and text live in a compatible embedding space, the model can transfer to new tasks without retraining a task-specific classifier.

Zero-shot intuition

At inference time, you can embed an image and several candidate text prompts such as “a photo of a dog” or “a photo of a bird,” then choose the text with highest similarity.

Exam takeaway

The course already asks about this objective directly. If a question asks what the loss is doing, say: maximize similarity of the correct pair and minimize similarity of incorrect batch pairings.

Common misunderstanding

CLIP is not trained to generate captions token by token. It is a contrastive representation learner, not an autoregressive captioning model.

Slides 31-33, 42 LLaVA-style pipeline Architecture

A multimodal assistant is usually a decoder LM wrapped around a pretrained vision encoder

The lecture’s LLaVA section gives the canonical recipe. First preprocess the image into patches. Then run a pretrained vision encoder such as a CLIP or SigLIP ViT to obtain a sequence of image vectors. Those vectors are linearly projected into the language model’s embedding dimension and inserted alongside ordinary text tokens. The decoder LM then continues doing what it already knows how to do: autoregressively generate text.

What the slides are saying

“Image tokens” are usually not raw pixels. They are vectors extracted by a pretrained visual backbone and then translated into the LM’s representational space.

Why the projector exists

The vision encoder and the LM were not trained with the same hidden size or embedding geometry. The projector is a learned adapter that lets the LM read the visual features.

Training detail worth remembering

When the lecture says “for image positions, skip the loss,” the point is that the model is trained to produce textual outputs, not to predict the image vectors themselves as next tokens.

Common misunderstanding

The presence of a language model does not mean the system has learned vision from scratch. In most practical multimodal systems, the heavy visual lifting is still done by the pretrained image encoder.

Slides 35-41 Molmo Practical design

Molmo shows that practical multimodal systems are constrained by token budget, not just model elegance

The most important Molmo slide is the image-encoding one. A 336 by 336 image with a ViT-L/14 encoder yields a 24 by 24 grid of features. Molmo then pools each 2 by 2 patch group, shrinking that to a 12 by 12 grid before projecting into the LM’s space. It also processes one full image and many crops, which gives the system both global context and local detail.

Why this is true

High-resolution vision features are useful, but an LM has a finite context budget. Pooling and crop selection are ways to spend that budget more intelligently.

Intuition

The full image answers “what is the scene?” while the crops answer “what small object or text might I miss if I only look globally?”

What the later Molmo slides imply

The dataset slides show that capability comes from task diversity too: captions, OCR-flavored questions, pointing, and synthetic data all teach different behaviors.

Exam takeaway

If asked how real systems differ from the clean lecture pipeline, a strong answer is: they compress, pool, crop, and curate data because raw visual tokens are too expensive.

Slides 43-44 Design lessons Ablation logic

The closing lesson is not “any vision interface works”; it is that pretrained visual structure and coordinated fine-tuning matter a lot

The PaliGemma ablations make two qualitative points. First, updating both the image encoder and the language model gives the best performance. Freezing or resetting important components hurts, which means the multimodal interface is not just a superficial wrapper around two independent pretrained models. Second, you can remove the image encoder and feed projected raw RGB patches directly to the LM, but doing so is much less sample-efficient.

Why this matters

A pretrained contrastive image encoder already knows useful visual abstractions. Throwing that away forces the LM to relearn vision from weaker supervision.

Best mental model

The image encoder provides compressed visual semantics; the LM provides linguistic fluency and instruction following. Strong multimodal systems need both parts to cooperate.

Exam takeaway

If a question asks what the ablations show, the safest answer is: pretrained vision features help, joint tuning helps, and raw patches are possible but data-hungry.

Common misunderstanding

“Raw patches can work” does not mean the encoder is unnecessary in practice. The slide’s point is about efficiency and optimization, not about proving that all designs are equally good.

Glossary

These are the terms most worth being able to define cleanly under time pressure.

Patch

A square chunk of the image. ViT uses patches as the visual analogue of tokens.

Patch embedding

The vector produced by flattening a patch and applying a learned linear projection.

Shared embedding space

A space where image and text vectors can be compared directly, typically by dot product or cosine similarity.

Contrastive loss

An objective that pulls matching items together and pushes mismatched items apart.

Image tokens

Continuous vectors representing visual content that are supplied to the language model.

Projector

The learned mapping that converts vision-encoder features into the LM’s embedding dimension.

Zero-shot classification

Classifying an image by comparing it to text prompts without training a dedicated classifier for that label set.

Sample efficiency

How much data a method needs to reach a given performance level. The raw-patch result is worse on this dimension.

Slides 4-12 Paradigm map Framing that matters

The lecture is not just surveying image generators; it is motivating a specific multimodal design goal

The early slides list four generative paradigms for images: autoregressive models, VAEs, GANs, and diffusion. The important hidden message is that the lecture is not trying to crown one universal winner. It is trying to solve a multimodal systems problem: if we want one transformer to handle both text and images, we need an image tokenizer to turn pixels into discrete symbols and an image de-tokenizer to turn those symbols back into pixels.

What the slides are saying

Autoregressive models generate one unit at a time, VAEs compress into a latent then decode, GANs learn to fool a discriminator, and diffusion learns to reverse a noising process. The lecture then focuses on the autoregressive route because it is the most naturally compatible with language-model style token prediction.

Why this matters

Lecture 11 only needed an image encoder because the output was text. Lecture 12 cares about image generation, so it needs both directions: image → tokens and tokens → image.

Intuition

The dream is “remove modality-specific bridges.” If text and image can both be represented as sequences of discrete symbols, one transformer can model them in a unified way.

Common misunderstanding

The lecture is not claiming autoregression is the best way to generate photorealistic images overall. It is claiming autoregression is the cleanest way to unify text and image under one token-prediction interface.

Slides 13-19 Attempt 1 Pixel autoregression

The most direct approach is to treat an image as a very long sequence of pixel values, but that simplicity becomes the main problem

The first attempt is pure language-model thinking: linearize the image into a sequence of pixel or channel values and train with maximum likelihood. If \(x_{\mathrm{img}} \in \mathbb{R}^{H \times W \times C}\), we convert it into a sequence \(x_1, \dots, x_T\) and model each element with \(p_\theta(x_t \mid x_{

The core recipe is just language modeling applied to pixels:

\[ x_{\mathrm{img}} \rightarrow x_1, \dots, x_T, \qquad x_t \in \{1, \dots, 256\}, \qquad \mathcal{L}_{\mathrm{MLE}} = \sum_{t=1}^{T} -\log p_\theta(x_t \mid x_{

For a \(1024 \times 1024\) RGB image, \(T \approx 1024 \cdot 1024 \cdot 3 = 3{,}145{,}728\) predicted values. That is why the sequence-length challenge is not a footnote; it dominates the design.

Why it struggles

Sequence length is enormous, and each individual pixel value carries very little semantic meaning compared with a word token. The model spends huge effort modeling low-level local detail before it can capture high-level structure.

Intuition

A single word can carry a concept like “bridge” or “zebra.” A single pixel mostly says “this one color channel is slightly brighter here.” That means semantic understanding is spread across many more positions.

What the example slides add

PixelRNN, Image Transformer, and iGPT show that the recipe is real and can scale, but the lecture groups them to make one point: practical pixel-level generation typically needs special attention patterns or very large compute because the raw sequence is too long.

Exam takeaway

If asked why raw-pixel autoregression is unattractive, say both pieces: it creates extremely long sequences and the tokens are semantically weak, so learning becomes expensive and data-hungry.

Slides 21-29 VAE bridge Latent-variable view

Before we can discretize images, the lecture briefly reviews the continuous-latent compression story behind VAEs

The VAE slides matter because VQ-VAE is easier to understand if you first remember what a standard VAE is doing. A VAE assumes data is generated from a latent variable \(z\): sample \(z \sim p(z)\), then decode it into an image through \(p_{\theta_{\mathrm{dec}}}(x \mid z)\). Training uses an encoder \(q_{\theta_{\mathrm{enc}}}(z \mid x)\) to approximate the posterior over latents and balances faithful reconstruction against keeping the latent space close to a simple prior.

The standard objective is:

\[ \mathcal{L}_{\mathrm{VAE}}(x) = -\mathbb{E}_{q_{\theta_{\mathrm{enc}}}(z \mid x)}[\log p_{\theta_{\mathrm{dec}}}(x \mid z)] + D_{\mathrm{KL}}\!\left(q_{\theta_{\mathrm{enc}}}(z \mid x)\,\|\,p(z)\right). \]

The first term says “reconstruct well.” The second says “do not let the encoder invent an arbitrary latent space that cannot be sampled from at generation time.”

Why the KL term exists

Without regularization, the encoder could memorize the training set in an awkward latent geometry. Matching the encoder distribution to a simple prior such as \(p(z)=\mathcal{N}(0,I)\) makes sampling and generation possible.

Missing context

The ELBO slide explains that this objective is a lower bound on \(\log p(x)\), and the reparameterization trick is what lets gradients flow through latent sampling during training. You do not usually need to reproduce the full derivation, but you should know what those phrases are doing there.

Why this is only a bridge

A standard VAE gives a continuous latent representation. That is already useful for compression, but it is not yet the discrete token sequence a language-model-style transformer wants.

Exam takeaway

If asked how a VAE differs from pure autoregression, the clean answer is: it learns a latent bottleneck and reconstructs from it, rather than predicting every observed variable directly one step at a time.

Slides 29-33 VQ-VAE Most testable Lecture 12 block

VQ-VAE turns “compress the image” into “pick entries from a learned visual vocabulary”

VQ-VAE keeps the encoder-decoder structure but changes the latent space from continuous to discrete. The encoder first produces a continuous vector \(z_e(x)\). A quantizer then compares \(z_e(x)\) to a learned codebook \(\{e_1, \dots, e_K\}\) and replaces it with the nearest codebook vector. The decoder reconstructs the image from that quantized representation, and the codebook index becomes a token that a transformer can model later.

The quantization step is the key equation to know:

\[ k^* = \arg\min_{j \in \{1,\dots,K\}} \lVert z_e(x) - e_j \rVert_2, \qquad z_q(x) = e_{k^*}. \]

This is why the model produces discrete image tokens: each local latent gets replaced by a codebook entry index.

Why the straight-through estimator is needed

The nearest-neighbor lookup uses an argmin, which is discrete and non-differentiable. The straight-through estimator lets decoder gradients act as if the quantization step were copied through, so the encoder still learns.

What the loss terms are doing

One term rewards good reconstruction. A second term moves codebook entries toward encoder outputs. A third “commitment” term discourages the encoder from wandering arbitrarily instead of settling near one codebook vector.

Intuition

A standard VAE says “compress into a point in continuous space.” VQ-VAE says “compress into a sequence of visual dictionary entries.” That makes the output much easier to hand off to a token model.

Common misunderstanding

The codebook indices are not human-interpretable class labels. They are learned discrete building blocks chosen because they help reconstruction, not because each one corresponds to a named semantic concept.

Slides 34-39 VQ-GAN Practical image tokens

VQ-GAN makes the discrete-token idea more visually useful by improving the tokenizer and then training a transformer on the resulting tokens

The lecture’s VQ-GAN section modifies the VQ-VAE objective by adding GAN-style and perceptual losses. The goal is not just to reconstruct the right average pixels, but to produce sharper, more realistic reconstructions that preserve perceptually important structure. Once the tokenizer and de-tokenizer are trained, we can tokenize a large image dataset and train a standard autoregressive transformer on the discrete image-token sequences.

The scaling benefit is the simplest thing to remember:

\[ 32 \times 32 = 1024 \qquad\text{versus}\qquad 256 \times 256 = 65{,}536. \]

If the tokenizer turns an image into a \(32 \times 32\) grid of codes, the transformer sees about sixty-four times fewer positions than raw \(256 \times 256\) pixels.

What VQ-GAN adds beyond VQ-VAE

VQ-VAE gives discrete latents; VQ-GAN tries to make those latents reconstruct images that look better to humans. The adversarial and perceptual terms target visual fidelity that plain reconstruction losses often miss.

Why this matters for multimodal modeling

A poor tokenizer is a poor vocabulary. If the code sequence throws away important visual detail, the downstream transformer can never recover it no matter how strong the language model is.

What the example-heavy slides imply

The output slides are there to show that better tokenizers matter. The lecture does not need you to memorize specific images; it wants you to see why perceptual quality of the tokenizer/de-tokenizer directly affects the final generative system.

Exam takeaway

If asked why VQ-GAN helps, say: it preserves the discrete-token interface of VQ-VAE but uses stronger image-quality objectives so the resulting tokens are a better substrate for transformer-based generation.

Slides 39-43 Unified models DALL-E and Chameleon

The endgame is a unified autoregressive model where text and image codes share one token-prediction framework

After learning a tokenizer and de-tokenizer, the lecture’s recipe is straightforward: add image tokens to the language model vocabulary, then train or fine-tune the model on sequences containing both text and image tokens. DALL-E and Chameleon are examples of this idea. The attraction is conceptual simplicity: one autoregressive model, one next-token objective, and a unified way to do multi-to-text, multi-to-image, or multi-to-multi generation.

Why this is elegant

Everything becomes next-token prediction. The model does not need one objective for text and a completely different one for images once both modalities share a discrete token interface.

Main cost

The tokenizer becomes a bottleneck. If it throws away information or introduces artifacts, the autoregressive model inherits that loss. This is the lecture’s main warning about the “everything is tokens” view.

Why Chameleon appears here

Chameleon shows this recipe is not just a historical curiosity. Even in 2024, large multimodal systems can still be built by first learning a visual tokenizer and then scaling a transformer over the resulting token stream.

Big-picture takeaway

The lecture closes by saying this approach is useful and conceptually clean, but not the final word on state-of-the-art image quality. That is why the final slide points toward diffusion next.

Glossary

These are the Lecture 12 terms most worth being able to explain quickly and precisely.

Image tokenizer

A model that maps an image into a sequence or grid of discrete codes so a transformer can model it as tokens.

Image de-tokenizer

The decoder that turns discrete image codes back into pixels, enabling actual image generation.

Latent variable

A hidden representation \(z\) that an encoder infers from the image and a decoder uses to reconstruct or generate it.

Prior

The distribution \(p(z)\) we want latent samples to follow so generation from sampled latents is possible.

Codebook

The learned set of vectors in VQ-VAE. Each encoder output is replaced by its nearest codebook entry.

Quantization

The step that replaces a continuous latent vector with a discrete codebook choice.

Straight-through estimator

A training trick that lets gradients pass through a discrete lookup as if it were continuous enough for backpropagation.

Perceptual loss

A loss that encourages reconstructions to preserve higher-level visual similarity, not just raw pixel-by-pixel accuracy.

Main ideas to keep straight
  • Lecture 12 is about generation, so it needs both an image tokenizer and an image de-tokenizer.
  • Raw-pixel autoregression is conceptually simple but suffers from massive sequence length and weak per-token semantics.
  • VAEs introduce the compression story: reconstruct well, but keep the latent space sampleable through a prior.
  • VQ-VAE discretizes the latent space with a codebook, which creates image tokens a transformer can model.
  • VQ-GAN improves the perceptual quality of those tokens, enabling stronger autoregressive token-based image generation.
  • Unified token models are elegant, but they inherit information loss from the tokenizer and are not the final answer for best image quality.

Slides 2-9 Diffusion setup Foundational

Diffusion replaces “predict the next token” with “iteratively denoise a corrupted sample”

The lecture’s first big move is conceptual, not algebraic. Lecture 12 tried to make image generation look like language modeling by discretizing images into tokens. Lecture 15 instead keeps images in a continuous space and says: define a fixed forward process \(q\) that gradually turns data \(x_0\) into noise \(x_T\), then learn a reverse process \(p_\theta\) that undoes those steps. This is why modern text-to-image systems feel so different from token-based autoregressive image models: they generate by iterative refinement, not by committing to one token at a time.

The Gaussian forward process and its direct-sampling shortcut are the key equations to know.

\[ q(x_t \mid x_{t-1}) = \mathcal{N}\!\left(x_t; \sqrt{\alpha_t}\,x_{t-1}, (1-\alpha_t)I\right), \qquad q(x_t \mid x_0) = \mathcal{N}\!\left(x_t; \sqrt{\bar{\alpha}_t}\,x_0, (1-\bar{\alpha}_t)I\right) \]

\[ \bar{\alpha}_t = \prod_{i=1}^{t}\alpha_i, \qquad x_t = \sqrt{\bar{\alpha}_t}\,x_0 + \sqrt{1-\bar{\alpha}_t}\,\epsilon, \qquad \epsilon \sim \mathcal{N}(0,I). \]

The schedule is chosen so that \(\bar{\alpha}_0 = 1\), \(\bar{\alpha}_T \approx 0\), and noise increases smoothly over time.

What the slides are saying

The forward process is fixed and easy to sample from; the reverse process is the thing we train. Keeping those roles separate is the first important mental checkpoint.

Why this matters

Training becomes a collection of easier denoising problems at different noise levels instead of a single impossible jump from random noise straight to a high-resolution image.

Missing context

The direct formula for \(x_t\) is what makes diffusion practical. It means training can jump to any timestep \(t\) in one shot instead of simulating every earlier noising step.

Common misunderstanding

The noise schedule is not learned from scratch in the basic lecture setup. What the model learns is how to reverse the chosen corruption process.

Slides 10-19, 22-25 Training and score view Most testable Lecture 15 block

The practical training rule is simple: sample a timestep, add known noise, and train the network to predict exactly that noise

The slides briefly derive training from an ELBO, decomposing it into prior matching, denoising, and reconstruction terms. But the lecture’s real punchline is that, under the Gaussian setup, the messy variational story collapses into a very clean objective: choose a random timestep \(t\), sample noise \(\epsilon\), construct \(x_t\), and regress onto the injected noise. That is the operational recipe students actually need to remember.

The conceptual ELBO split is:

\[ L = L_T + \sum_{t=2}^{T} L_{t-1} + L_0 \]

where \(L_T\) is prior matching, \(L_{t-1}\) is denoising, and \(L_0\) is reconstruction. In practice, the lecture reduces this to:

\[ L_{\mathrm{simple}}(\theta) = \mathbb{E}_{t, x_0, \epsilon}\!\left[\left\|\epsilon - \hat{\epsilon}_\theta(x_t, t)\right\|_2^2\right], \qquad x_t = \sqrt{\bar{\alpha}_t}\,x_0 + \sqrt{1-\bar{\alpha}_t}\,\epsilon. \]

Because the optimal predictor estimates the expected noise, the model also learns the score: \[ \hat{\epsilon}_\theta(z_t, t) \approx -\sigma_t \nabla_{z_t}\log p_t(z). \]

Why the \(\epsilon\)-loss is such a big deal

It turns the reverse step into plain supervised regression. The model is not asked to output a full image distribution directly; it is asked to identify the corruption that was applied.

Why predicting noise is especially useful

Quiz 4 highlights the key reason: the noise predictor is also a score estimator. That is what later continuous-time SDE and ODE formulations plug into.

Quiz-aligned timestep warning

If you oversample large \(t\) values, the model sees more heavily corrupted inputs. That usually helps coarse denoising but weakens learning on fine-detail steps near the data manifold.

Sampling loop to remember

At generation time, start from \(x_T \sim \mathcal{N}(0,I)\), then repeatedly use the denoiser inside the reverse Gaussian transition until you land near \(x_0\).

Slides 27-31 Guidance Conditional generation

Conditioning works by steering denoising toward samples that both look plausible and match the prompt

After the score interpretation is in place, conditional generation becomes much easier to describe. The lecture first introduces classifier guidance: add a classifier gradient that rewards samples likely to belong to class \(c\). It then moves to classifier-free guidance (CFG), which is the version used far more often in practice because it avoids training a separate classifier. The model is trained both with and without conditioning, and at sampling time the two predictions are combined to amplify the conditioning signal.

The CFG formula worth memorizing is:

\[ \tilde{\epsilon}_\theta(z_t, c) = (1+w)\epsilon_\theta(z_t, c) - w\epsilon_\theta(z_t). \]

The unconditional branch says “what looks like a valid image here?” The conditional branch says “what matches prompt \(c\) here?” The guidance scale \(w\) extrapolates in the prompt-following direction.

What the slides are saying

Classifier guidance adds an explicit classifier gradient; classifier-free guidance bakes the same idea into one diffusion model by contrasting conditioned and unconditioned predictions.

Why this matters

This is the mechanism behind the familiar “prompt adherence versus naturalness/diversity” tradeoff in modern text-to-image systems.

Intuition

The unconditional model keeps the sample on the manifold of realistic images. The conditioned model pulls the sample toward the requested concept. CFG leans harder in that conditioned direction.

Common misunderstanding

Bigger \(w\) is not automatically better. Too much guidance can reduce diversity, over-sharpen artifacts, or push samples toward brittle prompt satisfaction.

Slides 32-35 Extensions Practical system design

Once diffusion is viewed as score-based dynamics, the main engineering question becomes which bottleneck you want to attack

The “other extensions” table is one of the most useful compression slides in the whole deck. It says the major improvements are not random tricks. Latent diffusion attacks state size by moving diffusion into an autoencoder latent space. DiT changes the backbone from U-Net-style convolutions to a transformer. Deterministic samplers come from the continuous-time probability-flow ODE view. Distillation attacks the remaining bottleneck: too many denoising steps at inference time.

The continuous-time step that unlocks deterministic samplers is the probability-flow ODE:

\[ dz = \left[f(t)z - \frac{1}{2}g(t)^2 \nabla_z \log p_t(z)\right]dt. \]

It has the same marginals as the diffusion SDE, but it is deterministic, which is why ODE solvers and related fast samplers even make sense.

Why latent diffusion matters

Pixel space is too large and too expensive. Compressing into an autoencoder latent makes diffusion feasible at the resolutions modern image models care about.

Best way to remember the table

Latent diffusion reduces state size, DiT changes architecture, deterministic solvers reduce sampling randomness and steps, and distillation teaches a student to imitate many denoising steps in far fewer.

Missing context

The continuous-time score view is the bridge between “predict \(\epsilon\)” and “solve an ODE.” That bridge is why the lecture spends time on score functions at all.

Exam takeaway

If asked why predicting \(\epsilon\) is useful beyond the training loss, say: it gives the score term needed by score-based SDE/ODE samplers and therefore by many fast-diffusion variants.

Slides 36-40 Flow matching Modern direction

Flow matching changes the learning target from “undo this diffusion step” to “predict the velocity that moves samples from noise toward data”

The lecture’s last idea is not just “another sampler.” Flow matching changes the object the model learns. Instead of defining a stochastic reverse diffusion process and predicting noise, we define a path between an easy source distribution \(p_0\) and the data distribution \(p_1\), then train a model to predict the velocity field that carries samples along that path. The straight-line conditional path is the simplest case, and it is enough to connect the lecture to Stable Diffusion 3.

The straight-path conditional flow-matching equations are the ones to keep in memory:

\[ X_t = (1-t)X_0 + t x_1, \qquad \frac{d}{dt}X_t = x_1 - X_0, \]

\[ \mathcal{L}_{\mathrm{CFM}} = \mathbb{E}_{t, x_1, X_0}\!\left[\left\|v_\theta(X_t, t) - (x_1 - X_0)\right\|_2^2\right]. \]

The lecture’s key result is that this conditional objective gives the same optimum as the harder full flow-matching objective.

Why it can be faster

Flow matching learns a deterministic ODE with straighter paths, so numerical integration is often easier than reversing a long stochastic diffusion chain.

Exam comparison to keep straight

DDPM: stochastic SDE-style process, curved diffusion path, predicts noise, typically many steps. Flow matching: deterministic ODE, chosen path, predicts velocity, often fewer steps.

Stable Diffusion 3 tie-in

The case-study slide says SD3 combines flow-matching variants with latent-space modeling and strong text conditioning from CLIP and T5. The point is not to memorize the table values; it is to see where modern diffusion systems are heading.

Common misunderstanding

“Straight line” describes the chosen interpolation path in state space, not a promise that every perceptual trajectory is literally simple or that flow matching is automatically better in every setting.

Slides 2-12 Why RL LM motivation

RL appears here because imitation learning is not the same thing as optimizing task success

The lecture first recaps supervised fine-tuning and maximum likelihood, then names three problems with using that recipe as the whole story for advanced LM behavior. First, there is task mismatch: maximizing the probability of a likely response is not the same as maximizing helpfulness, correctness, harmlessness, or unit-test pass rate. Second, there is data mismatch: the available text can contain buggy, toxic, or otherwise undesired behavior, and the exact outputs we want are often scarce. Third, there is exposure bias: training conditions on gold prefixes, but test-time generation conditions on the model’s own mistakes. Slides 8-12 then reframe the learning loop: generate behavior, score it with a reward, and update the model toward the better trajectories.

What the slides are saying

RL replaces “imitate the provided output tokens” with “sample a trajectory, measure how good it was, and use that scalar feedback for learning.”

Why this matters

Many LM goals are sequence-level or task-level. A final answer can be correct even if it is not the most likely string in the dataset, and it can be wrong even if every local next-token choice looked plausible.

Missing context

RL does not remove the need for careful supervision. It shifts the burden from “provide exact target outputs” to “design or learn a reward signal that really captures what you want.”

Exam takeaway

If asked when RL is attractive, say: when the criterion we care about is easier to score at the trajectory level than to encode as token-level labels for every step.

Slides 14-20, 25-26 MDP formalism Foundational

The formal language is an MDP: states, actions, transitions, rewards, policies, and returns

After the motivation, the lecture switches to the standard RL abstraction: an agent interacts with an environment over time. At each step it receives information about the current situation, chooses an action, and gets a scalar reward. The environment then transitions to a new state. The full setup is summarized as a Markov Decision Process \((S, A, E, R)\), where \(S\) is the state space, \(A\) the action space, \(E(s_{t+1}\mid s_t, a_t)\) the transition dynamics, and \(R(s_t, a_t)\) the reward function. The agent acts according to a policy \(\pi(a_t\mid s_t)\).

The lecture’s main mathematical object is the trajectory and its return.

\[ \tau_{0:T} = (s_0, a_0, r_1, s_1, \dots, r_T, s_T), \qquad J(\pi) = \mathbb{E}_{\tau \sim d^\pi_\tau}\!\left[\sum_{t=1}^{T} r_t\right]. \]

The policy is good if the expected return under the trajectory distribution it induces is large.

What the symbols mean

Reward is the scalar feedback at one step. Return is the accumulated future reward along a trajectory. Confusing those two is one of the easiest ways to get lost.

Why the expectation matters

Even if the policy is fixed, the return is still random because the policy can sample actions stochastically and the environment can transition stochastically.

Algorithm map

The lecture briefly distinguishes policy-based, value-based, and model-based RL. It then deliberately narrows to policy-gradient methods, while noting that value functions often re-enter later as training stabilizers.

Common misunderstanding

The MDP is a modeling framework, not a guarantee that the problem is easy. Sparse rewards, delayed effects, and partial observability can still make learning very hard even when the notation is compact.

Slides 18, 23-24 Text generation as RL LLM translation

For language models, the state is usually the prompt plus the prefix, the action is the next token, and the reward is often sparse

The lecture’s most important example for this course is text generation. In the math-problem slide, the state is \(s_t = (x, y_{<t})\): the prompt plus the tokens generated so far. The action is the next token \(a_t = y_t\), the policy is the language model \(p_\theta(y_t \mid y_{<t}, x)\), and the environment simply appends the sampled token to the prefix. The reward can be very sparse, such as \(1\) only if the final answer is correct and \(0\) otherwise. That is why RL for LMs immediately becomes a credit-assignment problem: many intermediate tokens share one delayed verdict.

State versus observation

The lecture distinguishes environment state, observation, and agent state. In many LM examples the policy conditions on the visible prefix, so observation and agent state nearly coincide, but that is a modeling convenience, not a law of RL.

Intuition

A rollout is literally the model thinking out loud one token at a time. RL does not change that autoregressive interface; it changes how the whole trajectory is judged and updated.

Why sparse reward is hard

If only the final answer is scored, the algorithm has to decide which earlier actions deserve credit or blame. That is exactly why naive sequence-level policy gradient has high variance.

Exam takeaway

If asked to map LM generation into RL language, give all five pieces cleanly: state, action, policy, environment transition, and reward. The course likes this translation exercise.

Slides 28-31, 48-50 REINFORCE Most testable Lecture 16 block

REINFORCE uses the log-derivative trick to turn expected-return optimization into a sampled weighted log-prob update

The lecture now asks the obvious question: if the goal is to maximize expected return, why not just differentiate it and run gradient descent? The obstacle is that rewards are defined on sampled trajectories, which include discrete actions and possibly non-differentiable environment behavior. The score-function estimator solves this by moving the gradient onto the log-probability of the sampled outcome. Applied to trajectories, that gives the REINFORCE rule: sample a rollout from the current policy and weight each action’s log-probability gradient by the return that followed.

The score-function identity is the bridge from generic probability to policy gradient.

\[ \nabla_\theta \mathbb{E}_{x \sim p_\theta(x)}[f(x)] = \mathbb{E}_{x \sim p_\theta(x)}\!\left[f(x)\nabla_\theta \log p_\theta(x)\right]. \]

For RL, the lecture turns this into

\[ \nabla_\theta J(\theta) \approx \sum_{t=1}^{T} R_t \nabla_\theta \log \pi_\theta(\hat a_t \mid \hat s_t), \]

where \(R_t\) is the future return from that point onward. High-return actions get pushed up; low-return actions get pushed down.

Why the derivation works

The appendix’s key punchline is that trajectory probability factorizes into initial-state terms, policy terms, and environment-transition terms, but only the policy terms depend on \(\theta\). After taking the gradient, the environment drops out.

Best intuition

The algorithm is not asking “what is the gradient of the reward?” It is asking “which sampled actions made high reward more likely, so their probability should go up next time?”

Operational recipe

Collect trajectories, compute returns, build the loss \(\mathcal{L}_{PG} = -\sum_t R_t \log \pi_\theta(a_t \mid s_t)\), then backpropagate through the policy network.

Common misunderstanding

REINFORCE is unbiased in expectation, but it is usually too noisy to use alone at scale. The rest of the lecture is about fixing that variance and step-size problem without changing the core policy-gradient idea.

Slides 37-40 Variance reduction Quiz-aligned

Vanilla policy gradient is high variance, so the lecture reweights each step with discounted advantages instead of raw episode return

Once the lecture has vanilla REINFORCE on the table, it immediately asks how to make it less noisy. The first answer is discounting: later rewards are weighted by powers of \(\gamma\), which helps control how strongly distant outcomes affect earlier actions. The second answer is baselines: compare observed return to what was expected in that state, rather than to zero. The third answer is advantages: replace raw return with a better estimate of “how much better than expected was this action here?” The lecture then shows three common versions of that idea: a value-function baseline, a one-step TD residual, and generalized advantage estimation (GAE).

The core stabilization formulas are:

\[ R_t^{(\gamma)} = \sum_{t' = t+1}^{T} \gamma^{t' - t - 1} r_{t'}, \qquad \hat g_t = (R_t - b(s_t)) \nabla_\theta \log \pi_\theta(a_t \mid s_t). \]

\[ \hat A_t = R_t - v(s_t), \qquad \delta_t = r_t + \gamma v(s_{t+1}) - v(s_t), \qquad \hat A_t^{\mathrm{GAE}} = \sum_l (\gamma \alpha)^l \delta_{t+l}. \]

The unifying idea is that the gradient should care about how surprisingly good an action was, not just about the raw total reward of the whole rollout.

Why discounting exists

Delayed rewards create messy credit assignment. Discounting gives the algorithm a principled way to care less about outcomes that are farther away in time, which often reduces variance and changes the effective planning horizon.

Quiz-aligned baseline fact

Subtracting an action-independent baseline does not change the expected gradient because \(\mathbb{E}_{a \sim \pi}[\nabla_\theta \log \pi_\theta(a \mid s)] = 0\). What it changes is variance, not bias.

Why advantages help

A reward of 0.3 might be terrible on an easy prompt and great on a hard one. Advantage estimation normalizes by what was expected from that state, which makes updates more informative.

Common misunderstanding

The value function in this lecture is mostly a helper for estimating expected return and reducing policy-gradient variance. It is not replacing the policy or turning the method into pure value-based RL.

Slides 41-46 PPO LLM bridge

PPO controls step size by comparing the new policy to the old one and clipping overly aggressive changes

The lecture’s second stabilization idea is about how far the update moves the policy, not about how returns are estimated. Even with better advantages, policy-gradient updates are noisy, and a single large step can derail learning. Proximal Policy Optimization (PPO) handles this by comparing the new policy to the behavior policy that generated the data, forming a probability ratio, and clipping the objective once that ratio moves outside a safe range. The resulting training loop is simple and reusable: collect trajectories, compute advantages, compute a policy-gradient-style loss such as PPO, and update with SGD or Adam. That is why the final slides can immediately pivot to RLHF and RLVR: they mostly change the reward definition, while the policy-gradient plus PPO machinery stays recognizable.

The PPO quantities worth recognizing are the old-versus-new ratio and the clipped surrogate objective.

\[ \mathrm{ratio}(x,y) = \frac{p_\theta(y \mid x)}{p_{\theta_{\mathrm{old}}}(y \mid x)}, \qquad L_{\mathrm{PPO}} = \min\!\big( \mathrm{ratio}(x,y)A(x,y), \mathrm{clip}(\mathrm{ratio}(x,y), 1-\epsilon, 1+\epsilon)A(x,y) \big). \]

If the new policy tries to increase probability too much relative to the old one, the clipped term cuts off the incentive for even larger movement.

What clipping is doing

Clipping is a practical “do not trust one noisy batch too much” rule. It preserves the sign of the improvement signal while damping incentives for overly large policy shifts.

Why it matters for LLMs

LM RL often uses long, high-variance trajectories and imperfect rewards. PPO became popular because it keeps the familiar policy-gradient pipeline while making catastrophic jumps less likely.

How the closing slides fit

The RLHF and RLVR previews are there to show the reusable template: prompt-conditioned policy, scalar reward, an advantage or baseline, and PPO-style optimization. The reward changes; the optimization story mostly does not.

Exam takeaway

If asked how PPO differs from vanilla policy gradient, say: same policy-gradient core, but with explicit old-policy comparison and clipping to control update size.

Reward

The scalar feedback emitted at a particular step or after a trajectory.

Return

The sum of future rewards from a given point onward, possibly discounted by \(\gamma\).

Policy

The agent’s behavior rule, written as \(\pi(a \mid s)\). For LMs, this is the next-token distribution.

Trajectory / Episode

The state is the underlying situation relevant for decision-making; the observation is only the part visible to the agent.

Baseline

An estimate of expected return used to reduce variance by judging an action relative to what was expected.

Advantage

A signal measuring how much better or worse an action was than the expected outcome for that state.

PPO Clipping

A step-size control trick that limits how much the policy update can benefit from making the new policy too different from the old one.

Slides 2-9 Lecture 16 continuation Framework

Lecture 17 takes the Lecture 16 RL loop and asks which parts become design choices once the policy is a language model

The lecture opens by saying that the generic RL recipe from last class still applies: generate outputs, evaluate reward, and update the model. What changes is the modeling choice around the action space. In a one-step MDP, the model takes one action by generating a full response \(y\) for a prompt \(x\), then receives a sequence-level reward \(r(x,y)\). In a token-level MDP, the state is the prompt plus the current prefix \(s_t = (x, y_{<t})\), the action is the next token \(a_t = y_t\), and the reward can in principle be assigned token by token, though many LM settings still use only a terminal reward. The early slides also make the lecture’s real agenda explicit: initialization, prompt dataset, reward design, and policy-optimization details are the levers that define an RL-for-LLMs method.

The same policy can be written in either MDP view:

\[ \text{one-step: } s = x,\quad a = y,\quad \pi_\theta(a \mid s) = p_\theta(y \mid x),\quad r = r(x,y) \]

\[ \text{token-level: } s_t = (x, y_{<t}),\quad a_t = y_t,\quad \pi_\theta(a_t \mid s_t) = p_\theta(y_t \mid x, y_{<t}),\quad r_t \text{ often sparse}. \]

The point is not that one view is always right. The point is that they make different credit-assignment and engineering assumptions.

What the slides are saying

RL for LLMs is not a new family of math. It is Lecture 16’s policy-gradient story, plus explicit choices about what counts as an action, what data prompts are sampled from, and how the reward is obtained.

Why this matters

If the reward only exists for the completed response, the one-step view is often simpler. If intermediate tokens matter or token-level shaping is available, the token-level view can represent that more faithfully.

Missing context

The distribution you train on is generated by the current policy, not just by the original dataset. That means initialization and prompt selection are not peripheral details; they decide what behavior the policy gets to explore.

Exam takeaway

If a question asks for the “policy,” “trajectory,” or “reward source” in an LM RL pipeline, answer in LM terms: the policy is the next-token distribution, the trajectory is the sampled completion process, and the reward source is either a verifier or a learned model.

Slides 11-17, 28 RLVR Verifiable rewards

Verifiable-reward tasks are the lecture’s cleanest setting because the reward comes from a checker rather than a learned critic

The first example, reversing a string, is intentionally simple. It lets the lecture separate the RL mechanics from the harder question of how to define reward. The model is usually supervised-fine-tuned first on \((x,y)\) pairs, then further optimized with RL under a reward that is easy to verify: \(1\) if the reversal is correct, \(0\) otherwise. The slides then generalize the same idea to math answers and code that passes tests. This is why recent terminology uses Reinforcement Learning with Verifiable Rewards (RLVR): the reward is supplied by an external checker for correctness, not by humans rating helpfulness and not by a learned reward model guessing human taste.

The reward in RLVR is usually something directly checkable, such as:

\[ r(x,y) \in \{0,1\} \text{ for correctness,} \qquad r(x,y) = \text{fraction of passed tests for code.} \]

This is why reversal, math with a checkable final answer, and coding with unit tests appear together in the slides.

Why SFT comes first

The “cold start” idea is that RL works better once the model already knows the task format. SFT moves the policy into a region where high-reward samples are at least plausible, which reduces wasted exploration.

Why RLVR is attractive

It avoids the biggest RLHF problem: imperfect learned rewards. If correctness is mechanically checkable, the optimization target is much less ambiguous.

Hidden limitation

RLVR only works cleanly when the task has a reliable checker. Many alignment or conversational goals do not, which is why the lecture later turns to preference models.

Common misunderstanding

RLVR does not mean “no RL algorithm is needed.” It only means the reward source is verified externally. You still need advantages, a loss, and stable optimization.

Slides 19-28 GRPO Quiz-calibrated

Group-based advantages say “judge each sampled answer relative to the other answers for the same prompt,” then optimize with PPO

The reversal example then turns into the lecture’s main optimization idea. For each prompt \(x^{(i)}\), sample \(K\) outputs from the current policy, score them, and compare each reward to the group average. That gives a relative advantage signal without training a separate value function. The lecture’s terminology slide summarizes the recipe as GRPO: a PPO-style loss plus group-based advantages, often with group standard-deviation normalization. This is best understood as a direct continuation of Lecture 16’s baseline story. A learned value baseline estimates “what reward was expected here?” using another model; a group-based baseline estimates that same idea using sibling samples from the current policy on the same prompt.

The group-relative advantage is:

\[ A^{(i,k)} = \frac{r^{(i,k)} - \mathrm{mean}(r^{(i,1)}, \dots, r^{(i,K)})}{Z}, \]

where \(Z\) is often \(1\) or the group standard deviation. The policy update is then fed into a PPO-style clipped objective using those advantages.

Why this is useful

It avoids fitting a value model, which is especially appealing when the reward is sparse and you already plan to sample multiple completions per prompt.

Lecture 16 connection

This is still baseline subtraction. The baseline is just prompt-local and sample-based instead of being predicted by a learned critic.

Main tradeoff

You pay for the simpler baseline by generating multiple outputs for each prompt, and the approach gives less direct guidance about how to assign credit inside long trajectories before the final reward is known.

Quiz-calibrated warning

Because sequence-level rewards are often broadcast across all output tokens, extra length normalization can bias training toward shorter or longer answers. Quiz 4’s bonus question suggests the course expects you to reason about that side effect even if the slides do not dwell on it.

Slides 30-35 Math reasoning RLVR case study

Math reasoning keeps the RLVR template but makes the sparse-reward and credit-assignment issues much more visible

The DeepSeek-R1 case study is presented as the same basic recipe on a harder task. The prompt is a math problem, the output is a chain of thought plus an answer, the MDP is still treated as one-step at the sequence level, and the reward combines a checkable final-answer signal with format constraints. The data slide also makes an important broader point: once a task ends in a mechanically checkable output, the same setup can extend beyond pure math to code, STEM multiple choice, logic, and other domains. The plots are not there for memorizing benchmark values. Their conceptual message is that RL can improve final accuracy and can also change how the model searches, for example by producing longer traces and revising intermediate reasoning.

What the slides are saying

Use a pretrained model, sample full reasoning traces, score them with a verifier, compute group-based or PPO-style updates, and keep a KL-style anchor so the model does not drift too chaotically.

Why this connects to Lecture 16

The policy is still the LM, the trajectory is still a sampled completion, and the reward is still delayed. The only difference is that the environment now checks mathematical correctness instead of a game score.

Important caution

Longer chain-of-thought is evidence that the policy changed, not proof that the reward captured genuine reasoning quality. If the checker is weak, the model can still optimize the wrong thing.

Exam takeaway

If asked why RL can help math despite SFT already knowing the domain, say: RL changes the sampling distribution toward trajectories that end in correct answers, which can improve search and self-correction under sparse terminal rewards.

Slides 37-48 RLHF pipeline Alignment

When correctness is not mechanically verifiable, the lecture switches from RLVR to RLHF: build a reward proxy, then optimize the policy against it

The alignment part of the lecture starts with the question that RLVR cannot answer: how do we reward a chat completion for being helpful, safe, truthful, or honest? One option is a direct assessment model that scores a prompt-response pair directly. Another is a preference model, motivated by the fact that humans often find pairwise comparison easier than assigning an absolute scalar score. RLHF packages this into a three-stage pipeline: first supervised fine-tuning on demonstrations, then reward-model training on chosen-versus-rejected responses, then RL that samples new outputs and optimizes the policy against the learned reward. The large-scale Tulu 3 slide reinforces that this is a data-engineering pipeline as much as an optimization recipe: choose prompts, generate candidate responses, annotate preferences, then train reward and policy models on top of that data.

Why it is a pipeline

The reward is not directly given by the environment. It has to be approximated from demonstrations and comparison data before PPO or any other RL algorithm can optimize it.

What is learned directly

The policy learns to imitate demonstrations in stage 1, the reward model learns to rank responses in stage 2, and stage 3 only indirectly transfers those preference judgments into policy behavior. Stages 1-2 are mostly offline data fitting; stage 3 becomes online again because the current policy must generate fresh samples to optimize.

Preference data source

The lecture explicitly allows either human labelers or a stronger LM to provide rankings. The important point is still the same: preference labels supervise the reward proxy, not the policy tokens directly.

Exam comparison

RLVR and RLHF differ first in reward source, not in the optimizer. Both can use PPO-style updates; RLHF just needs extra steps because the reward itself is learned.

Slides 49-52 Bradley-Terry Directly quiz-tested

Pairwise preference learning says the probability that one answer is preferred should depend on the difference between their reward scores

The reward-model slides answer the “where does this loss come from?” question instead of asking you to memorize it blindly. Given a prompt \(x\) and two responses \(y^+\) and \(y^-\), the reward model \(r_\phi(x,y)\) should assign a higher score to the preferred response. The Bradley-Terry model turns those scores into a comparison probability: if better responses should win pairwise comparisons more often, then the probability of preferring \(y^+\) over \(y^-\) should be a sigmoid of the score difference. Maximizing the likelihood of observed comparisons therefore becomes the familiar negative log-sigmoid loss.

The lecture’s reward-model derivation is:

\[ p(y^+ \succ y^-) = \frac{e^{r_\phi(x,y^+)}}{e^{r_\phi(x,y^+)} + e^{r_\phi(x,y^-)}} = \sigma\!\big(r_\phi(x,y^+) - r_\phi(x,y^-)\big). \]

\[ \mathcal{L}_{\mathrm{RM}} = - \sum_{(x, y^+, y^-) \in \mathcal{D}} \log \sigma\!\big(r_\phi(x,y^+) - r_\phi(x,y^-)\big). \]

Only the score difference matters, which is why the loss is pairwise rather than “classify each response independently.”

Why pairwise labels are popular

People are often more consistent at saying “A is better than B” than at assigning a calibrated numerical score to each answer separately.

Why the sigmoid appears

The Bradley-Terry parameterization exponentiates latent scores and normalizes them into a two-item probability, which algebraically collapses into a sigmoid over the score difference.

Hidden assumption

The model assumes there is an underlying scalar utility that explains pairwise preferences reasonably well. Real human judgments can violate that assumption, which is one reason reward models are imperfect.

Common misunderstanding

Do not write this as two separate binary-classification losses on \(y^+\) and \(y^-\). The quiz answer key and the lecture both emphasize that the correct loss depends on the difference \(r(y^+) - r(y^-)\).

Slides 53-60 Reward hacking KL-constrained PPO

RLHF adds a KL anchor because maximizing an imperfect reward model too aggressively invites reward hacking

The lecture’s reward-hacking slides are the conceptual heart of RLHF. If the reward model only imperfectly reflects the real objective, the policy can discover outputs that score well under the model while being bad for users. The simplest classroom example is a toxicity or offensiveness reward that can be “won” by always returning an empty string. The standard mitigation in the lecture is a reference-policy KL penalty: maximize expected learned reward, but pay a cost for moving too far from the SFT policy \(p_0\). Operationally, the slides explain this in three equivalent ways: as a constrained objective, as an extra reward term using \(\log \frac{p_\theta}{p_0}\), and as a closed-form optimal policy proportional to the reference policy times an exponentiated reward.

The lecture’s regularized objective is:

\[ \max_\theta \; \mathbb{E}_{x,y \sim p_\theta}[r_\phi(x,y)] - \beta D_{\mathrm{KL}}(p_\theta(\cdot \mid x) \parallel p_0(\cdot \mid x)). \]

Using a sample \(\hat y \sim p_\theta(\cdot \mid x)\), the KL term is approximated by

\[ \log \frac{p_\theta(\hat y \mid x)}{p_0(\hat y \mid x)}, \qquad r'(x,\hat y) = r_\phi(x,\hat y) - \beta \log \frac{p_\theta(\hat y \mid x)}{p_0(\hat y \mid x)}. \]

The policy update is then usually implemented with PPO plus an advantage estimator such as GAE.

Why reward hacking happens

The reward model is only a proxy. RL is powerful enough to find distribution shifts or odd outputs that fool the proxy without actually improving the real objective.

What the KL term is doing

It keeps the RL-updated policy near a known-good reference policy, usually the SFT model, so reward optimization cannot roam arbitrarily far in search of proxy exploits.

Lecture 16 connection

The optimizer is still recognizable from the previous lecture: PPO plus advantages. What changed is the reward source and the need for regularization against the reference model.

Common misunderstanding

KL regularization does not make the reward model correct. It only reduces how aggressively the policy can exploit its mistakes. A better reward model and better preference data still matter.

Glossary

These are the Lecture 17 terms most worth being able to define quickly and precisely.

One-step MDP

An LM RL formulation where the action is a full response and the reward is computed at the sequence level.

Token-level MDP

An LM RL formulation where the state is the prompt plus prefix and the action is the next token.

Cold Start / SFT before RL

Supervised fine-tuning done before RL so the model already knows the task format and can reach rewarding behaviors more easily.

RLVR

Reinforcement Learning with Verifiable Rewards: RL where correctness is checked by an external verifier such as an answer checker or unit tests.

GRPO

A PPO-style optimization recipe using group-based advantages, typically by comparing multiple sampled responses for the same prompt.

Preference Model / Reward Model

A learned scorer \(r_\phi(x,y)\) trained so preferred responses get higher scores than rejected ones.

Bradley-Terry Loss

The pairwise preference objective that uses \(\sigma(r(y^+) - r(y^-))\) to model which answer should be preferred.

Reference Policy

The older or SFT policy \(p_0\) kept around so RLHF can penalize drifting too far from it.

Reward Hacking

Improving the learned reward signal without improving, and sometimes while hurting, the real task objective.

KL Penalty

A regularizer that charges the policy for moving too far from the reference distribution, usually weighted by \(\beta\).

End-of-Lecture Summary

Main ideas to keep straight
  • Lecture 17 turns Lecture 16’s generic RL language into LM-specific design decisions about action granularity, reward source, and initialization.
  • RLVR is the clean setting where reward comes from a verifier, so string reversal, math answers, and coding tests can all share one RL template.
  • GRPO is best understood as an advantage-estimation choice on top of PPO-style RL, not as a separate reward source.
  • Math-reasoning RL uses the same sequence-level policy and sparse reward story as Lecture 16, but with checkable final answers.
  • RLHF is a three-stage pipeline: SFT, reward modeling from preferences, then RL against the learned reward.
  • Because learned rewards can be exploited, RLHF usually adds a KL penalty to keep the policy near a reference model.
Exam-oriented comparison
  • RLVR: defined by its reward source. It uses a checker, not a learned reward model.
  • RLHF: defined by its pipeline and reward source. It needs preference data, a learned reward model, and usually PPO plus KL control.
  • PPO: a loss / update-control mechanism. It can appear in both RLVR and RLHF.
  • GRPO: an advantage recipe. It can sit on top of different reward sources, and it is especially natural when multiple samples per prompt are already being generated.
  • The lecture’s own summary table is organized by reward, advantages, and loss. That is the safest way to keep these methods from collapsing into one vague bucket.
